# ROADMAP — дорожная карта строительства

Принцип: **сначала весь фронтенд на мок-данных, потом бэкенд, потом крипта.** Каждая фаза имеет
критерии выхода. Не переходи к следующей фазе, пока текущая не закрыта. Отмечай выполненное прямо
здесь (`- [x]`).

Почему такой порядок:
- **UI/UX дешевле всего менять, когда за ним ничего нет.** Прокликиваем и вылизываем продукт
  целиком, прежде чем заплатить сложностью сети и инфраструктуры.
- **Фронт не одноразовый**, потому что пишется против `DataProvider` (см. `CLAUDE.md` §3). Бэкенд и
  крипта подключаются как реализации интерфейса, экраны не трогаются.

---

## Фаза 0 — Скелет и дизайн-система

**Цель:** репозиторий, в котором можно сразу строить экраны, и язык дизайна, из которого они растут.

- [ ] Next.js (App Router) + TypeScript + Tailwind, строгий tsconfig
- [ ] Подключить шрифты и токены из `frontend/design-system.md` (CSS-переменные + tailwind config)
- [ ] Базовый layout: тема dark-first, контейнеры, сетка, типографические утилиты
- [ ] Примитивы из Radix + обёртки (Button, Input, Dialog, Tabs, Tooltip, Toast) в фирменном стиле
- [ ] `DataProvider` интерфейс + пустой `MockDataProvider` + хук `useData()` + ENV-переключатель
- [ ] Каркас навигации: маршруты-заглушки под все экраны из `frontend/screens.md`
- [ ] Линт/формат/CI, Storybook **или** страница `/dev/kitchen-sink` со всеми компонентами

**Критерии выхода:** проект запускается; видна тема; есть пустой роутинг по всем экранам; компоненты-
примитивы отрендерены на kitchen-sink; `useData()` возвращает мок (пусть пока пустой).

## Фаза 1 — Фронтенд на мок-данных (главная фаза по UX)

**Цель:** полностью рабочий продукт «на ощупь» — все экраны, все состояния, все потоки — но данные из
`MockDataProvider`. Никакого бэкенда, никакой сети, никакой крипты. Можно прокликать насквозь.

Реализуй `MockDataProvider` целиком по контракту из `frontend/mock-data.md` (фикстуры, искусственная
задержка, детерминированные ответы, симуляция модерации/успеха доната).

Экраны (полная спека — `frontend/screens.md`):

- [ ] **Auth / Connect** — подключение кошелька (мок) + Sign-In, экран сессии
- [ ] **Channel page (вид донатера)** — донат-виджет, разбивка комиссии 97/3, лидерборд, тиры,
      «моя репутация на этом канале», лента показанных сообщений
- [ ] **Donate flow** — сумма, опц. текст, превью комиссии, подтверждение, момент финальности,
      состояние «текст ушёл на модерацию (HELD)»
- [ ] **Viewer profile / My standing** — тир, очки, бейдж, прогресс до следующего тира, история донатов
- [ ] **Light profile settings** — display_name, avatar, bio, links (опциональный профиль)
- [ ] **Streamer dashboard** — обзор канала, метрики, последние донаты
- [ ] **Moderation queue** — HELD-сообщения, авто-флаги, «Показать / Скрыть», карантин
- [ ] **Channel settings** — репутационная формула (linear / sublinear / bracket), множители, decay;
      тиры/бейджи/цвет ника/перки; минимумы доната; лимит длины и профанити-политика; режим имён;
      режим показа текста; оверлей/алерты; модераторы
- [ ] **Channel activation** — одноразовый сбор (~$2), что он разблокирует, состояние «активирован»
- [ ] **Channel block-list** — канальные блокировки кошельков
- [ ] **Overlay view** — отдельный публичный вид для OBS: алерты, показанные сообщения, тир-бейджи
- [ ] **Discovery / Explore** — публичный индекс активированных каналов (минимально)
- [ ] **Operator / T&S console** — саспенд/бан каналов и кошельков, карантин нелегальщины,
      санкционный скрин, инцидент-лог, лестница наказаний (спека — `frontend/screens.md`)
- [ ] **Системные состояния** — empty, loading (skeletons), error, disabled, забаненный канал

Потоки (спека — `frontend/user-flows.md`):

- [ ] Донат с текстом → HELD → стример «Показать» → SHOWN на оверлее
- [ ] Донат с текстом → HELD → «Скрыть» → деньги и репутация остаются, текст не публикуется
- [ ] Накопление репутации → переход тира → визуальный момент тир-апа
- [ ] Базовый канал пытается принять донат-с-текстом → отклонение (тир не активирован)
- [ ] Флуд/абьюз → канальный блок → лестница наказаний оператора

**Критерии выхода:**
- Каждый экран существует во всех состояниях (empty/loading/error/success).
- Каждый поток из `user-flows.md` проходится насквозь на моках.
- Все инварианты §4 из `CLAUDE.md` отражены в UI (например: после доната — явный «финально»;
  репутация не имеет кнопок «передать/продать»; текст по умолчанию скрыт).
- Доступность: фокус с клавиатуры, контраст, `prefers-reduced-motion`, мобильная вёрстка.
- Продукт можно показать человеку и он поймёт его, не читая спеку.

> Это точка для ревью UX. Здесь имеет смысл собрать обратную связь и итерации **до** написания бэкенда.

## Фаза 2 — Бэкенд (оффчейн), `ApiDataProvider`

**Цель:** заменить мок настоящими данными, не трогая экраны. Полная спека — `backend/spec.md`.

- [ ] Схема БД по `docs/data-model.md` (профили, журнал репутации, конфиги, сообщения, баны)
- [ ] **Репутационный движок** как чистая детерминированная функция от журнала + конфига (инвариант §4.4)
- [ ] Версионирование и хэширование конфига стримера; банкинг очков на момент доната
- [ ] API-поверхность (REST/tRPC) ровно под методы `DataProvider`
- [ ] Модерационный конвейер: локальный wordlist → детект языка → авто-слой → роутинг → человек
- [ ] Очередь сообщений, переходы HELD → SHOWN/HIDDEN, карантин, дедуп по хэшу контента
- [ ] Баны/саспенды/блок-листы, лестница наказаний, инцидент-лог + preservation
- [ ] `ApiDataProvider` реализует интерфейс; флаг `NEXT_PUBLIC_DATA_SOURCE=api`
- [ ] Сидинг dev-данными, эквивалентными мок-фикстурам (чтобы UI выглядел так же)

**Критерии выхода:** все экраны Фазы 1 работают на `ApiDataProvider` без изменений в компонентах;
репутация перевычисляется детерминированно; модерация гоняется на мультиязычном тест-миксе.

## Фаза 3 — Крипта (ончейн), `ChainDataProvider`

**Цель:** реальные деньги и реальная репутация из ончейна. Полная спека — `crypto/spec.md`.

- [ ] Подключение кошелька (`@solana/wallet-adapter-react`) + Sign-In-With-Solana (gasless auth)
- [ ] Построение донат-транзакции: одна tx, расщепление 97% стример / 3% трежери, memo-атрибуция
- [ ] Создание ATA стримера в той же транзакции при отсутствии; учёт газа/priority fee; газовый флор
- [ ] Индексер: ловит инфлоу в трежери, парсит memo, кормит репутационный движок
- [ ] Сбор активации канала (~$2 в трежери), активация по подтверждению
- [ ] Опц. ончейн-хэш текста (неизменность пруфа)
- [ ] `ChainDataProvider` / гибрид: чтение репутации из индексера, отправка доната через кошелёк
- [ ] Тестнет (devnet) → калибровка анти-флуда, кривых, decay, retention
- [ ] **Перед мейннетом:** живая юридическая консультация в выбранной юрисдикции (`docs/legal-and-risk.md`)

**Критерии выхода:** донат проходит на devnet донор → стример напрямую, оператор денег не касается;
репутация начисляется из индексера и совпадает с детерминированным пересчётом; все инварианты §4 живы.

## После ядра — мини-игры (вне этой версии)

Журнал репутации (`docs/data-model.md`) уже резервирует отрицательные дельты и не-донатные типы
событий (`DISPUTE_*`, `GAME`, `REFUND`), чтобы первая игра — «задание-донат с проверкой комьюнити»
(whitepaper v1.1) — подключилась модулем, не переписывая фундамент. Это отдельный документ и
отдельный профиль юридического риска. Не в этом репозитории.

## Сквозные нерешённые вопросы (тянуть через все фазы)

- **Мастер-переменная:** юрисдикция инкорпорации + допуск пользователей из США. Влияет на геофенс и
  на то, что вообще можно строить. Решается до мейннета.
- Калибровка анти-флуда: размер сбора активации, скорость карантина/бана, rate-limit, retention.
- Калибровка репутации: кривая (linear / bracket), decay — на тестнете.
- Политика бана: только креатор-роль или полный бан кошелька.
- Глубина ревью-восстановления для ложных срабатываний.
