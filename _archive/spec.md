# Фронтенд — спецификация (Фаза 1)

Цель Фазы 1: **полностью рабочий продукт «на ощупь» на мок-данных.** Все экраны, все состояния, все
потоки — без бэкенда, без сети, без крипты. Это точка, где вылизывается UI/UX до того, как платить
сложностью.

Связанные документы: `design-system.md` (язык), `screens.md` (экраны), `components.md` (библиотека),
`user-flows.md` (потоки), `mock-data.md` (контракт данных). Механика — `../docs/core-spec.md`.

---

## 1. Принципы

1. **Данные — только через `DataProvider`.** Ни один компонент не зовёт `fetch`/RPC/Solana. См. §3 ниже
   и `CLAUDE.md` §3. Это то, что делает фронт-первый-подход не одноразовым.
2. **Инварианты ядра видимы в UI.** Финальность доната, непередаваемость репутации, приватность текста
   до показа — это не только про бэкенд, это должно *чувствоваться* в интерфейсе (`CLAUDE.md` §4).
3. **Два регистра не путать.** Денежные элементы — спокойные/точные; статусные — заработанные/живые
   (`design-system.md` §1).
4. **Состояния — первый класс.** Каждый экран существует в empty / loading / error / disabled / success.
   Мок специально умеет их воспроизводить.
5. **Честный копирайт.** От лица пользователя, без обещаний справедливости, финальность — прямо.

## 2. Информационная архитектура (роли и маршруты)

Три роли (могут совмещаться на одном кошельке): **донатер**, **стример (креатор)**, **оператор (T&S)**.

```
/                         Discovery / Explore (публичный индекс каналов)
/c/[handle]               Страница канала (вид донатера) — главный публичный экран
/c/[handle]/overlay       Оверлей для OBS (публичный, без хрома)
/me                       Профиль зрителя: standing по каналам, история донатов
/me/profile               Настройки лёгкого профиля (opt-in)

/studio                   Дашборд стримера (обзор)
/studio/queue             Очередь модерации (HELD → показать/скрыть)
/studio/settings          Настройки канала (формула, тиры, минимумы, имена, показ, оверлей, модераторы)
/studio/activation        Активация канала (одноразовый сбор)
/studio/blocklist         Канальные блокировки

/ops                      Консоль оператора / T&S (саспенд/бан, карантин, санкции, инцидент-лог)

/connect                  Подключение кошелька + Sign-In (модалка/экран)
/dev/kitchen-sink         (dev) все компоненты и токены
```

Навигация: публичная шапка (лого, поиск каналов, «Подключить кошелёк» / адрес+бейдж). В `/studio` —
сайдбар студии. В `/ops` — отдельный admin-хром. Оверлей — без всякого хрома.

## 3. Контракт с данными (`DataProvider`)

Весь доступ к данным — через интерфейс из `mock-data.md`. Обёртка:

```ts
// провайдер выбирается по ENV; компоненты этого не знают
const provider = createDataProvider(process.env.NEXT_PUBLIC_DATA_SOURCE); // "mock" | "api" | "chain"

// в компонентах — только хук поверх TanStack Query
const { data, isLoading, error } = useChannel(handle);
const donate = useDonate(channelId);
```

Все запросы — через TanStack Query (кэш, loading/error из коробки). Мутации (донат, show/hide, апдейт
конфига) — через `useMutation`, с оптимистичными апдейтами там, где уместно.

`MockDataProvider` обязан:
- держать фикстуры (`mock-data.md`) в памяти;
- эмулировать задержку (например 150–600ms) и иногда ошибку (флаг для тестирования error-стейтов);
- эмулировать переходы: донат → запись + событие репутации; текст → HELD; show/hide меняет состояние;
  накопление очков → пересчёт тира; и т.д.;
- быть детерминированным (seeded), чтобы скриншоты/ревью воспроизводились.

## 4. Форматирование (единые утилиты)

- `formatUSDC(micro: bigint): string` → `"$12.50"` (моно, tabular-nums).
- `formatPoints(p: number): string` → `"5,000"` (моно).
- `shortAddress(a: string): string` → `"7xKp…3fQa"` (+ копирование по клику).
- Время — относительное («3 мин назад») с тултипом на абсолютное.
- Никогда не форматировать деньги/очки вручную в компонентах — только через утилиты.

---

## 5. Доступность и адаптив

- Донат-флоу обязан работать с телефона (донор часто мобильный).
- Студия и оверлей оптимизированы под десктоп, но не ломаются на узком экране.
- Клавиатурный фокус, ARIA на модалках/табах/тултипах, `prefers-reduced-motion` (`design-system.md` §5).

---

## 6. Критерии готовности Фазы 1

См. `ROADMAP.md` → Фаза 1. Короткий тест: продукт можно дать человеку, и он поймёт его без спеки;
каждый поток из `user-flows.md` проходится насквозь; каждый экран есть во всех состояниях.
