> M-1…M5 (§9): **M-1…M2 ПОСТРОЕНЫ (2026-07-05)** — журнал/канон чтения/споры на канистре, эскроу-
> RESOLVER = тресхолд-адрес; журнал исполнения — `migration-plan.md`. Мейннет ⏸; следующая — M3.
> Поглотила субстрат спеки голосования G3b (ADR 0020; сам файл спеки удалён при чистке доков —
> экономика при реализации упрощена решениями владельца: без депозитов, кворум фикс, табло

> github.com/dfinity/sol-rpc-canister. Курс: 1T циклов = 1 XDR ≈ $1.35.

## 0. Идея в одном абзаце

Solana остаётся **костью №1 — деньгами** (донаты 97/3, активация, эскроу — не трогаем ни байта
в донат-пути). Всё, что дорого или невозможно считать на Solana — журнал репутации, веса,
голосование, вердикты споров — переезжает в **кость №2: канистру на Internet Computer**,
которая сама наблюдает за Solana (SOL RPC canister, консенсус 3-из-N провайдеров) и сама
подписывает вердикты тресхолд-ключом (Ed25519, ключа не существует целиком нигде). Наш сервер
сжимается до **кожи**: модерация, приватные тексты, операторская консоль — всё, что и должно
оставаться централизованным. Фронт переезжает на asset-канистру (сертифицированная раздача).
Пользователи не платят никогда и нигде; наши издержки — десятки долларов в месяц.

```
КОЖА      наш сервер: модерация (OpenAI), приватные тексты, /ops, инциденты   ← централизовано, законно
             │ хэши текстов ▲ показ/скрытие
КОСТЬ №2  ICP-канистры: журнал+репутация, конфиг-параметры, споры/голоса,     ← трастлесс-логика,
          вердикты (tEd25519 → Solana), фронт (asset-канистра)                  дёшево, юзер не платит
             │ наблюдает (SOL RPC, 3-из-N) ▲ подписывает resolve
КОСТЬ №1  Solana: донаты 97/3 + memo, активация, эскроу-PDA                   ← деньги, не трогаем
```

## 1. Модель доверия — честно (главное изменение против v1.1)

| Уровень | Кто охраняет | Класс доверия |
|---|---|---|
| Кость №1 (деньги) | консенсус Solana, тысячи валидаторов | максимальный |
| Кость №2 (правда/правила) | подсеть ICP: 13 узлов (обычная) / 34 (fiduciary, там же тресхолд-ключи), известные операторы; governance — NNS (DAO) | высокий, но мягче кости №1 |
| Кожа | мы | доверенный, ограничен правилами слоёв |

Иерархия: **Solana L1 > подсеть ICP > мультисиг > наш сервер.** Компрометация 2/3 подсети =
компрометация тресхолд-ключа. Почему размен приемлем — **радиус поражения ограничен нашими же
существующими замками**:

1. Деньги НЕ переезжают: канистре нечего украсть (донаты идут донор→стример, эскроу в PDA).
2. Резолвер остаётся **bounded** (ESC-1): тресхолд-адрес может лишь выбрать сторону СПОРНОГО
   эскроу — получатели зашиты в PDA, перенаправить деньги третьему нельзя даже при полном
   сговоре подсети.
3. `resolve_timeout` остаётся permissionless: смерть/зависание ICP-части = деградация ровно до
   сегодняшнего поведения (таймер-презумпция). Деньги не застревают никогда.

Худший случай сговора подсети — неверные исходы спорных эскроу. Не кража, не заморозка.
Governance-хвост (NNS может обновлять подсети) — задокументированное остаточное доверие.

## 2. Кость №1 — Solana (что остаётся и что меняется)

**Не меняется:** донат-транзакция (2×transferChecked 97/3 + memo — священный путь), активация,
аттестация payout (H1), клейм-модель эскроу, все ESC-инварианты, `resolve_timeout`.

**Меняется одно:** в эскроу-программе `RESOLVER` = **тресхолд-Ed25519-адрес канистры споров**
(вместо devnet-ключа оператора). Совмещается с обязательным mainnet-редеплоем (пин mint ESC-7,
прод-окна ESC-9). Кнопки резолвера в UI и env `ESCROW_RESOLVER` умирают.

**Что НЕ строим из спеки v1.1 (устарело с переездом субстрата):** Solana-программа счётчиков
репутации, чекпоинты в PDA, слоты голосов, шардинг табло, пачки Ed25519-проверок, банк спора
с возмещением подателям — вся эта инженерия существовала, чтобы уместить логику в дорогой
Solana-аккаунтинг. В канистре она не нужна.

## 3. Кость №2 — канистры ICP

### 3.1 `core`-канистра («хребет») — Rust

Одна канистра держит всю трастлесс-логику (масштабирование — §8):

1. **Индексер Solana** — таймер каждые N секунд → SOL RPC canister (`getSignaturesForAddress`
   по трежери-ATA, `getTransaction`, только `finalized` — наш M2-принцип переносится буквально;
   детерминизм finalized-ответов дружит с консенсусом outcalls). Разбор пары 97/3 + memo —
   порт нашего `extractDonation` (та же чистая логика). Курсор — в stable memory.
2. **Журнал репутации** — append-only в stable memory (до 500 GiB на канистру; наш журнал —
   сотни байт на донат: миллион донатов ≈ сотни МБ ≈ $0.1–0.5/мес хранения). Свёртка = порт
   `computePoints`; история весов — просто журнал (никаких чекпоинт-структур: выборка «вес на
   момент T» = свёртка по ts ≤ T, дёшево при 2B инструкций/с).
3. **Конфиг-параметры канала** (governance-часть: `minReputationToDispute`, `minWeightToVote`,
   кворум-коэффициент, окна, D_max) — пишутся ТОЛЬКО по ed25519-подписи владельца канала
   (тот же приём, что аттестация payout), изменения вступают с таймлоком ≥ полного цикла
   задания (замок §8.9 спеки голосования). Косметика (тиры, описания) сюда НЕ входит — кожа.
4. **Споры и голосование** — ПОСТРОЕНО в M2, экономика упрощена решениями владельца
   (2026-07-05, yellow-paper §18.5-8b/8c): **без депозитов/пошлин/залогов** (наказание — только
   репутация: −50 за активный проигрыш, +10 за выигрыш; анти-спам — порог репутации на
   открытие), **кворум — фикс от стримера** (governance `quorum_micro`, дефолт 1 очко; √-формула
   убрана), снимок веса на момент открытия. Голос = ed25519-подпись Solana-кошелька, летит в
   канистру напрямую (ingress ≈ $0.000004 за голос — платим мы, юзер ничего). Проверка подписи —
   обычный код внутри канистры. **Табло и голоса открыты живьём** (решение владельца; конверты
   commit-reveal для споров ≥ $50 — мейннет-план). Продажа голоса — принятый остаток (v1.1 §8.5).
   Детали реализации — yellow-paper §7.3.
5. **Вердикт → Solana**: финализация по таймеру канистры → сборка `resolve_dispute(to_streamer)`
   → **тресхолд-подпись Ed25519** (≈ $0.036/подпись) → `sendTransaction` через SOL RPC.
   Нюанс отправки: свежий blockhash плохо проходит консенсус провайдеров → **durable nonce**
   (официально рекомендованный обход) — заводим nonce-аккаунт для резолвер-адреса.
6. **Экспорт и доказуемость**: query-методы (бесплатные, мгновенные) + **certified data** —
   ответы канистры сертифицированы подсетью, читатель проверяет подлинность криптографически,
   не доверяя даже шлюзу. `verify-export` эволюционирует: сверка «канистра == Solana» вместо
   «наш сервер == Solana». Пруф-якорь в Solana оставляем как дешёвую кросс-чейн страховку
   (дайджест журнала канистры раз в сутки, ~копейки).
7. **Хэши текстов** — канистра хранит ТОЛЬКО хэши (memo.m, task_id-коммитменты). Сами тексты —
   никогда (приватность §4.6 + снимаемость/комплаенс: state канистры видят узлы подсети и
   его нельзя гарантированно «изъять»).

### 3.2 `assets`-канистра — фронт

- Статическая раздача SPA с **сертификацией ответов** (шлюз/браузер проверяют, что контент
  отдан сетью нетронутым — UI нельзя тихо подменить даже нам, только через управляемый
  контроллером апгрейд). Кастомный домен + автоматический TLS — штатно.
- **Ограничение, честно:** канистры не исполняют серверный JS — **SSR невозможен**, только
  SPA/статика. Наш Next.js App Router использует серверные роуты → фронт конвертируется в
  чистый SPA (все наши страницы уже «use client», данные — через `DataProvider`; серверные
  route-handlers переезжают: трастлесс-часть → canister API, кожа → наш API). Практически:
  либо `next export`-совместимая перестройка, либо переезд на Vite+React Router при миграции —
  решение на M3, экраны не переписываются (архитектурный замок ADR 0001 окупается).
- Рекомендуемый объём ассетов ≤ ~1 GiB (наш бандл — мегабайты).

### 3.3 `IcpDataProvider` — четвёртая реализация DataProvider

Экраны не знают о миграции (CLAUDE.md §3): новый провайдер = гибрид — трастлесс-чтения и
голоса/споры → канистра (agent-js, queries бесплатны и быстры), тексты/модерация/операторка →
наш API как сейчас, деньги → кошелёк как сейчас. Личность остаётся кошельком Solana: каждое
значимое действие несёт ed25519-подпись (канистре безразличен транспортный principal).

## 4. Кожа — наш сервер (сжимается, не исчезает)

Остаётся централизованным намеренно (требование владельца + комплаенс):

| Остаётся у нас | Почему не в канистре |
|---|---|
| Тексты сообщений/заданий (HELD→SHOWN, карантин) | приватность до показа + гарантированная снимаемость; state канистры видят узлы и не сотрёшь по требованию |
| Модерационный конвейер (OpenAI, LLM-судья) | недетерминированные ответы несовместимы с консенсусом outcalls; приватный ключ API |
| Операторская консоль /ops, инциденты, санкции | это и есть законная централизованная власть (trust-layers §4.2) |
| Косметика каналов (тиры/описания/перки), профили, дискавери, нотификации | кожа по определению, менять свободно |
| Продуктовая аналитика | не влияет на истину |

Бан/тейкдаун по-прежнему НЕ трогают деньги и число репутации — теперь это физика двух костей,
а не наша вежливость.

### 4.1 Может ли кожа тоже стать канистрой? (вопрос владельца, разобрано 2026-07-02)

Да, но не целиком — «централизованная канистра» (контроллер = мы) — нормальный паттерн, и
бОльшая часть кожи переезжает чисто: операторская консоль, санкции/инциденты, косметика каналов,
профили, дискавери. Это кандидат на фазу **M6** (после основной миграции): плюсы — не нянчить
VPS, репликация/аптайм из коробки, один деплой-стек; финансово ≈ нейтрально.

Два куска сопротивляются по проверенным причинам:

1. **LLM-модерация.** HTTPS-outcalls канистр реплицированы (каждая реплика делает вызов, ответы
   должны сойтись в консенсус) — недетерминированные ответы OpenAI это ломает; общепринятый
   обход — idempotent-proxy (кэширующий прокси, чтобы все реплики увидели ОДИН ответ)… который
   сам является Web2-сервером. Плюс API-ключ в канистре в момент вызова видим операторам узлов.
   Итог: модерационный конвейер остаётся нашим микросервисом (stateless-«лямбда»: precheck →
   LLM → вердикт) — сервер не исчезает, но сжимается с «весь бэкенд» до одной функции.
2. **Plaintext приватных текстов и CSAM-карантин.** vetKeys позволяют хранить тексты в канистре
   ШИФРТЕКСТОМ (узлы видят только шифртекст; расшифровка — клиентски у стримера/модераторов),
   но гарантия действует «до момента расшифровки»: любая обработка plaintext ВНУТРИ канистры
   (авто-модерация, карантин) экспонирует его злонамеренной реплике. Плюс карантин/preservation
   нелегальщины на репликах в разных юрисдикциях — юридически мутнее, чем наш подконтрольный
   сервер. Тексты-at-rest в канистре через vetKeys — интересное v2-исследование, обработка —
   только у нас.

Принцип при любом раскладе: **skin-канистра не получает write-доступа к core-канистре** —
кожа и кость №2 разделены контроллерами и интерфейсами, слои не смешиваются.

## 5. Экономика (проверенные тарифы, июль 2026)

| Статья | Цена | Наш масштаб |
|---|---|---|
| Хранение (13-узловая подсеть) | ~$0.45/GiB/мес | журнал за годы — сотни МБ → **центы/мес** |
| Приём голоса (ingress ~1 КБ) | ~$0.000004 | 100 000 голосов ≈ **$0.4**, платим мы |
| Тресхолд-подпись Ed25519 | ~$0.036 (34-узл. подсеть ключа) | 1–2 подписи на спор → **копейки** |
| HTTPS-outcall (SOL RPC) | ~$0.0001–0.0005/вызов (базовая ~49M циклов + байты) | опрос каждые 20 с ≈ **$10–40/мес** — главная статья |
| Исполнение update-вызова | ~$0.000007 базово | пренебрежимо |
| Фронт (assets + трафик) | в тех же циклах | **единицы $/мес** |

Итого вся кость №2 — **порядка десятков долларов в месяц**, оплачивается циклами с нашего
кошелька (reverse-gas: пользователь не платит никогда). Операционная обязанность: держать
канистры заправленными (freezing threshold — канистра замирает при пустом балансе; мониторинг
в runbook, авто-пополнение). Анти-спам ingress: ранняя дешёвая проверка (подпись + вес ≥ порога
из журнала) до тяжёлой работы; rate-limit по адресу.

## 6. Что умирает, что упрощается (сверка со спекой v1.1)

| Было (вариант Б, Solana-программа) | Стало (вариант В, канистра) |
|---|---|
| Программа счётчиков + обёртка донат-tx (CPI) | **донат-путь нетронут**; канистра наблюдает |
| Чекпоинты в PDA + компакция | журнал в stable memory, свёртка на любую дату |
| Слоты голосов, рента, overflow | нет аккаунтов — нет ренты |
| Пачки по 4–8 подписей, Ed25519-precompile, податели, возмещения, банк спора на доставку | голос летит напрямую; банк спора остаётся только как ЭКОНОМИКА (депозит/пошлина — анти-спам и «проигравший платит»), не как касса на газ |
| Шардинг табло (write-contention) | не нужен |
| Нагрузочная репетиция 100k на Solana | репетиция на канистре (дешевле и проще) |
| Резолвер = PDA программы голосования | резолвер = тресхолд-адрес канистры (тот же bounded) |

Экономические правила при реализации M2 (решения владельца, 2026-07-05): наказание только за
активный проигрыш (+10/−50), D_max, фиксация параметров с таймлоком, локальность репутации —
перенесены; **депозит-пошлина+залог и √-кворум — ОТМЕНЕНЫ** (без денежных наказаний; кворум —
фикс от стримера), табло открыто. Каталог атак §8 спеки остаётся ориентиром с поправкой на эти
решения; специфичные для Solana-субстрата пункты (8.12 осушение банка микро-пачками) заменяются
ICP-аналогами (спам-ingress → §5).

## 7. Новые риски (честный список)

1. **Молодость Solana-интеграции ICP** (Chain Fusion→Solana ~2025): durable nonce для отправки
   tx; только IPv6-провайдеры у SOL RPC; «быстро меняющиеся ответы» (свежий blockhash) требуют
   обходов. M0-прототип обязан прогнать это на живом devnet до любых обязательств.
2. **Подсеть = 13/34 известных узлов + NNS** — см. §1. Фиксируем в trust-layers как «кость №2».
3. **Второй стек**: Rust-канистры, dfx, циклы, другие аудиторы. Смягчение: канистра портирует
   НАШУ ЖЕ логику (extractDonation, computePoints, машина споров) — она уже написана и
   оттестирована на TS, порт со сверкой выходов 1:1 (golden-тесты на общих векторах).
4. **Cycles-опека**: пустой баланс = заморозка канистры. Runbook: мониторинг + авто-top-up;
   `resolve_timeout` страхует деньги даже при заморозке.
5. **SPA-фронт**: теряем SSR/серверные роуты Next (наши экраны и так клиентские; SEO для
   app-страниц не критичен, лендинг можно держать где угодно).
6. **Приватность стейта канистры**: узлы подсети видят память → в канистре нет ни текстов, ни
   чего-либо приватного (§3.1-7) — только то, что и так публично или хэши.
7. **Двойная бухгалтерия на переходе обязательна** (M0–M2): канистра и наш сервер ведут журнал
   параллельно, `verify-export` сверяет ТРИ источника (Solana ↔ канистра ↔ сервер) — расхождение
   останавливает миграцию.

## 8. Масштабирование

Один `core` — на годы вперёд: 500 GiB stable memory, ~1000 сообщений/блок на подсеть, 2B
инструкций/с на поток; наш профиль (донаты + споры) — смешные объёмы. Точки роста, заложенные
в дизайн, но не строящиеся сейчас:

- **Шардирование по каналам** (actor-model: канистра-на-N-каналов, реестр-роутер) — если один
  канал станет «национальным телевидением».
- Вынос индексера в отдельную канистру (чистое разделение «наблюдатель/судья»).
- Archive-канистры для старых сегментов журнала (паттерн ICP ledger).

## 9. Фазы миграции (каждая — со светофором владельца)

> Детальный план исполнения (репо-структура, golden-паритет, откаты, что нужно от владельца) —
> `docs/migration-plan.md`.

**Принцип непрерывности (требование, не надежда):** миграция незаметна для пользователей.
Кошелёк/вход — те же (Solana-подписи, никаких ICP-кошельков от людей); каналы, репутация и
история сохраняются — канистра пересобирает журнал из ПЕРВОИСТОЧНИКА (история донатов на цепи),
а не из нашего экспорта; донат-флоу байт-в-байт прежний; домен и экраны те же. Сайт не
останавливается: каждый кусок переключается только после зелёной сверки трёх источников
(Solana ↔ канистра ↔ сервер), до M2 всегда есть откат.

## 10. Открытые вопросы

1. Rust-канистра: один монолит `core` vs `core`+`indexer` с самого начала.
2. Фронт: Next static-export vs переезд на Vite (решить на M3 по фактическому объёму боли).
3. Конверты для мелких споров: одношаговый режим (state скрыт от публики самой канистрой) —
   достаточно ли, или всюду двухфазно.
4. Пруф-якорь кость№2→кость№1 (суточный дайджест журнала канистры в Solana-memo): включать ли
   с M0 (дёшево, добавляет кросс-чейн проверяемость).
5. Судьба текущего `indexer-service`/PGlite после M2: сервер держит только кожу — упрощение
   схемы БД (тексты/инциденты/профили), журнал уходит.
6. NNS/subnet-риски: следить за развитием (размер подсетей, decentralization roadmap ICP).
7. **M6 — skin-канистра** (§4.1): перенос нетекстовой кожи в централизованную канистру;
   модерационный микросервис остаётся; vetKeys для текстов-at-rest — исследование.
