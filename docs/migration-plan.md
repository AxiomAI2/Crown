# План миграции на канистры — исполнение архитектуры v3

> Детализация фаз M0–M5 из `canister-architecture.md` (ADR 0021) до уровня «можно начинать».
> Написан 2026-07-02, реализация не начата. Правила наследуются: непрерывность для пользователей
> (архитектура §9), деньги трогаем последними, светофор владельца на каждом шаге.

---

## 0. Три правила чистоты (весь план стоит на них)

1. **Golden-паритет.** Канистра не «пишется заново» — она ПЕРЕПИСЫВАЕТ нашу уже оттестированную
   TS-логику (разбор доната, свёртка очков, машина споров). Из TS выгружаются тест-векторы
   (JSON: вход → ожидаемый выход), и Rust-код обязан проходить **те же самые векторы**
   байт-в-байт. Расхождение = стоп. Это превращает «переписали на другой язык» из риска в
   механическую проверяемую операцию.
2. **Переключение только по зелёной сверке.** Каждый кусок истины переезжает в три шага:
   канистра ДУБЛИРУЕТ (сервер — канон) → сверка горит зелёным N дней подряд → канистра
   СТАНОВИТСЯ каноном (сервер — дублёр/выключен). Ревизор `verify-export` сверяет три источника:
   Solana ↔ канистра ↔ сервер.
3. **Деньги — последними.** Редеплой эскроу (M2) начинается только после того, как M0–M1
   доказали чтение цепи, стабильность канистры и тресхолд-подпись на живом devnet.

## 1. M-1 «Подготовка» (до любого кода канистр)

**Репозиторий** — монорепо расширяется, ничего не ломая:

```
standing/
├── anchor/            Solana-программа (как есть)
├── canister/          НОВОЕ: Rust-workspace
│   ├── core/          core-канистра (индексер, журнал, споры)
│   ├── core.did       candid-интерфейс — публичный контракт канистры (аналог IDL)
│   └── dfx.json
├── testdata/golden/   НОВОЕ: тест-векторы из TS (см. ниже)
└── src/ …             как есть
```

**Golden-векторы** — новый скрипт `scripts/export-golden.ts` выгружает из существующих тестов
и живых devnet-данных:

- `donations.json` — реальные parsed-транзакции devnet → ожидаемый разбор (`extractDonation`);
- `reputation.json` — журналы → ожидаемые очки (`computePoints`/`computePointsAsOf`, включая
  дробные, клампы, asOf-границы);
- `disputes.json` — сценарии машины споров (все переходы, кворум, tally, репут-эффекты).

Rust-тесты канистры читают эти же файлы. `npm run golden && cargo test` — один светофор паритета.

**Тулчейн:** `dfx` (SDK ICP), Rust, **PocketIC** (локальный IC для интеграционных тестов —
стандарт DFINITY; есть тестовые тресхолд-ключи). Локальный дев-стенд: `dfx start` + канистра,
смотрящая на настоящий Solana devnet через outcalls.

**От владельца:** кошелёк циклов (~$20–30 в ICP токенах — хватит на месяцы M0–M2) + строка в
runbook про его пополнение.

**Выход из M-1:** пустая канистра деплоится локально и на mainnet ICP; golden-векторы
выгружены; `cargo test` зелёный на заглушках.

## 2. M0 «Канистра-наблюдатель» (ничего не решает, всё сверяется)

**Строим:** core-канистра v0:
- таймер → SOL RPC canister → подписи трежери-ATA → `getTransaction` (finalized) → разбор
  донатов/активаций (Rust-порт `extractDonation`, проверенный golden'ами) → журнал в stable
  memory; курсор там же;
- query-API: standing, лидерборд, журнал канала, экспорт (+ certified data);
- тресхолд-Ed25519: канистра получает свой Solana-адрес, подписывает ТЕСТОВУЮ memo-транзакцию
  на devnet через durable nonce — доказываем весь контур подписи ДО того, как он понадобится
  для денег.

**Сверяем:** `verify-export.ts --canister <id>`: журнал сервера == журнал канистры == цепочка.
Гоняется в CI/вручную; целевой критерий — **зелёный 7 дней подряд** при живом трафике devnet.

**Откат:** выключить канистру. От неё ничто не зависит.

**Светофоры владельца:** новая ✅-строка в ревизоре; отчёт с реальными цифрами (латентность
outcalls, цена циклов/день, подпись прошла).

## 3. M1 «Канистра — канон чтения»

**Строим:**
- `IcpDataProvider` (пятая реализация DataProvider, гибрид как ChainDataProvider): репутация/
  лидерборды/журнал — из канистры (agent-js, query-вызовы бесплатны и быстры); всё остальное —
  как раньше. Включается env-флагом; экраны не меняются.
- Ончейн-конфиг канала в канистре: governance-параметры (пороги, кворум-коэффициент, окна,
  D_max) пишутся подписью владельца канала, изменения — с таймлоком. Студия получает секцию
  «Параметры споров» (подпись кошельком, как аттестация payout).
- Сервер продолжает вести свой журнал (дублёр) — сверка живёт.

**Откат:** флаг обратно — фронт читает сервер.

**Светофоры:** цифры в UI не изменились после переключения; сверка зелёная; параметры канала
меняются подписью и ждут таймлок.

## 4. M2 «Споры» — первое и единственное касание денег

**Строим:**
- Модуль споров в канистре: `open_dispute` (депозит: пошлина+залог), приём голосов-подписей
  (формат из спеки, ed25519-проверка), конверты для ≥$50, табло, финализация по таймеру,
  репут-эффекты (+10/−50, только активный проигрыш), D_max, √-кворум. Всё по
  `games/onchain-voting-spec.md` v1.1 — экономика уже спроектирована и отревьюена.
- **Редеплой эскроу-программы**: `RESOLVER` = тресхолд-адрес канистры. Одним релизом с
  mainnet-пинами ESC-7 (mint) и ESC-9 (прод-окна) — три давно висящих пункта закрываются разом.
- Красивая деталь контракта, которая делает переход бесшовным: `resolver` записывается в каждый
  эскроу **при fund** — старые лётные эскроу доживают со старым резолвером, новые рождаются с
  канистрой. Миграции живых денег НЕ существует как задачи. Откат симметричен: редеплой с
  прежней константой — новые эскроу вернутся к старой схеме, канистровые доживут у канистры
  (плюс вечный страховочный `resolve_timeout`).
- Кнопки резолвера и `ESCROW_RESOLVER` удаляются из UI/env.

**Депозит спора:** на devnet — devnet-USDC в эскроу-подобный PDA ИЛИ упрощённо учётом в
канистре; решение на входе в M2 (открытый вопрос спеки №1).

**Светофоры:** сценарий спора из manual-testing тремя кошельками — исход приходит из канистры
без нашего участия; смоук-скрипт `dispute-smoke` (аналог escrow-smoke) против живой связки;
нагрузочная мини-репетиция (10k синтетических голосов).

## 5. M3 «Фронт на asset-канистре»

- Конверсия в SPA: решение Vite+React Router vs `next` static export — по фактическому объёму
  боли (экраны уже клиентские, DataProvider абстрагирует данные; переезжают только серверные
  route-handlers: трастлесс → канистра, кожа → наш API).
- Asset-канистра + наш домен (регистрация в HTTP-gateway, TLS автоматом), сертифицированная
  раздача.
- Кожа-API остаётся на сервере (модерация, тексты, /ops) — фронт ходит в оба.

**Откат:** DNS обратно на старый хостинг.

## 6. M4 «Governance» и M5 «Мейннет»

- M4: контроллеры core-канистры → мультисиг; политика апгрейдов с таймлоком; процедура
  публична. (Blackhole/SNS — по зрелости, отдельное решение.)
- M5: полный аудит (эскроу-редеплой + core-канистра — бюджетируется $10–50k), нагрузочная
  репетиция 100k голосов, юр-мастер-переменная, mainnet-деплой. Kожа-канистра (M6, §4.1
  архитектуры) — после всего, опционально.

## 7. Операционка (дельта в runbook по ходу фаз)

- Циклы: мониторинг балансов канистр + автопополнение; freezing threshold = сайт-риск №1 новой
  эпохи (деньги пользователей НЕ страдают — `resolve_timeout`, но индексация встанет).
- Деплой канистры: `dfx deploy` + запись версии/hash в ADR-лог апгрейдов.
- Ключи: cycles-кошелёк владельца — в тот же ряд, что treasury/anchor (runbook «Ключи»).
- Сверка: `verify-export --canister` в ежедневный ритуал (потом CI).

## 9. Чего в плане НЕТ намеренно

- Переписывания экранов (DataProvider-замок), миграции пользователей (непрерывность §9),
  миграции лётных эскроу (resolver per-escrow), изменения донат-пути (священен), переноса
  модерации/текстов (кожа остаётся, §4.1), больших взрывных релизов (каждая фаза — маленькая,
  со своим откатом).
