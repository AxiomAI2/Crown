# Ручное тестирование — плейбук для не-программиста

> Как проверить систему, не читая код. Почти все золотые инварианты (CLAUDE.md §4) **видны глазами**:
> деньги дошли или нет, текст скрыт или нет, цифра сошлась или нет. Этот документ — сценарии
> «шаги → ожидаемо → баг, если», плюс три команды-светофора, плюс честная карта того, что руками
> проверить нельзя (и что это покрывает вместо тебя).
>
> Запуск стенда и грабли — `docs/runbook.md`. Нашёл баг — неси сценарий + что ожидал + что увидел.

---

## 0. Подготовка (один раз, ~15 минут)

**Нужно три роли** — заведи в Phantom три аккаунта (Add account) и подпиши их прямо в кошельке:

| Роль | Кто это | Как завести |
|---|---|---|
| **Стример** | создаёт канал, принимает донаты | обычный новый аккаунт |
| **Донор** | донатит, шлёт задания, спорит | обычный новый аккаунт |
| **Оператор** | консоль /ops | импортируй ключ из `.treasury-devnet.json` (Import private key; ключ публичен — ТОЛЬКО devnet) |

Дальше:
1. Phantom → Settings → Developer Settings → **Testnet mode / Solana Devnet**.
2. SOL на газ обоим кошелькам — [faucet.solana.com](https://faucet.solana.com) (может лимитить — 429, пробуй позже).
3. Devnet-USDC донору — [faucet.circle.com](https://faucet.circle.com), сеть Solana Devnet.
4. `npm run dev` → http://localhost:3000.
5. **Чистый мир**: останови сервер → удали папку `.data/` → запусти снова. Делай это перед большим прогоном.

Удобно: два браузера (или обычное+инкогнито окно) — в одном стример, в другом донор, чтобы не
перелогиниваться. Переключение аккаунта в Phantom автоматически переключает сессию на сайте.

## 1. Команды-светофоры (копипаст, читаются без кода)

Запускаются в терминале из папки проекта. Читать просто: **все ✅ / passed — порядок; любой ❌ / FAIL — неси мне.**

| Команда | Что доказывает |
|---|---|
| `npm test` | 116 автотестов: математика репутации, крипто-коммитменты, стейт-машина эскроу, модерация. Ответ «Tests 116 passed» = вся математика, которую ты не проверишь руками, сходится |
| `npx tsx scripts/verify-export.ts --channel <handle> --chain` | **независимый ревизор**: пересчитывает репутацию каждого донора из журнала, проверяет подпись адреса выплат, сверяет дайджесты с блокчейном. Все ✅ = «сервер не подкручивает цифры» — это проверка инварианта §4.4 одной командой |
| `npm run typecheck && npm run lint` | внутренняя целостность кода. Пустой вывод = ок |
| `npm run golden && (cd canister && cargo test)` | **паритет миграции на канистры** (с M-1): Rust-версия логики даёт байт-в-байт те же ответы, что рабочая TS-версия, на общем эталоне `testdata/golden/`. «7 passed» = порт не разошёлся с продом |
| `npx tsx scripts/verify-export.ts --channel <handle> --canister <url>` | **сверка трёх источников** (с M0, при поднятом локальном стенде канистры — runbook «Канистры ICP»): журнал канистры (пересобран ИЗ ЦЕПОЧКИ) == журнал сервера, донат-в-донат. ✅ «записей совпали» = канистра-наблюдатель видит те же деньги, что и сервер |
| `npx tsx scripts/dispute-smoke.ts` | **спор без площадки** (с M2, ~5–7 мин, нужен стенд канистры): настоящий эскроу на devnet проходит спор целиком — открытие и голос подписями кошельков, вердикт и ончейн-резолв тресхолд-ключом канистры, возврат денег донору, +10 репутации. Финальное «✅ DISPUTE-SMOKE ПРОЙДЕН» = исход спора не зависит от нас |

После КАЖДОЙ пачки правок от ИИ прогоняй первую и третью. Вторую — когда на стенде есть живой канал.
Четвёртую — когда правка касалась миграции на канистры (папка `canister/`) или файлов
`reputation.ts` / `chain/indexer.ts` / `escrow-task/machine.ts`.

## 2. Сценарии: деньги (инварианты §4.1 некастодиальность, §4.2 финальность)

### 2.1 Обычный донат
1. Донором открой `/c/<handle>` → введи 1 USDC → «Задонатить» → подтверждение.
2. **Смотри в окно Phantom ПЕРЕД подписью** — там видно, куда идут деньги: два перевода,
   0.97 на адрес стримера и 0.03 на трежери. Это некастодиальность, видимая глазами:
   платформа в получателях ОТСУТСТВУЕТ.
3. После подписи — «подтверждается…» ~15–30 секунд (это нормально: ждём финализации сети), затем
   момент «Готово. Деньги ушли стримеру.»
4. Открой ссылку на Solana Explorer из карточки доната — транзакция реальна, суммы совпадают.
5. Баланс USDC стримера в Phantom вырос ровно на 0.97.

**Баг, если:** в кошельке при подписи больше двух переводов или другие адреса; «Готово» без транзакции
в Explorer; баланс стримера не сошёлся; появилась любая кнопка «отменить/вернуть донат» (денег
возврата в ядре НЕТ — это инвариант, а не недоработка).

### 2.2 Донат ниже минимума / без кошелька
- Сумма ниже минимума канала → кнопка не активна, подпись даже не запрашивается.
- Без подключённого кошелька → форма предлагает подключить, не даёт «донатить в пустоту».

## 3. Сценарии: репутация (§4.3, §4.4, §4.5)

### 3.1 Курс и дробление
1. Задонать 2.5 USDC → «моя репутация» на канале выросла ровно на **2.5 очка** (1 USDC = 1 очко).
2. Задонать двумя донатами по 0.5 → суммарно ровно +1.0 (дробление не даёт ни больше, ни меньше).

### 3.2 Непередаваемость — проверка отсутствием
Пройди все экраны: НИГДЕ не должно быть кнопок «передать очки», «продать», «обменять», «подарить
репутацию». Их отсутствие — юридический замок §4.3. Появились — баг критический.

### 3.3 Перевычислимость — «сервер не решает, а считает»
1. Внизу страницы канала — ссылка «Скачать журнал и пересчитать» → открывается JSON (это публичный
   журнал всех донатов канала).
2. Прогони ревизора: `npx tsx scripts/verify-export.ts --channel <handle> --chain` → все ✅.
   Это значит: цифра на лидерборде = честная сумма журнала, журнал совпадает с блокчейном.

### 3.4 Оператор не трогает число
Забань кошелёк донора оператором (сценарий 6.3) → открой профиль донора: очки НА МЕСТЕ (число
честное, просто бесполезное — донор ничего не может делать). Очки исчезли/уменьшились — баг §4.5.

### 3.5 Канон чтения из канистры (режим icp, миграция M1)

1. Подними стенд канистры (runbook «Канистры ICP») и включи режим icp (там же). Открой канал —
   лидерборд и твой стендинг теперь приходят В БРАУЗЕР прямо из канистры, мимо нашего сервера.
   Проверка глазами: останови канистру (`dfx stop`) → плашки репутации покажут ошибку сети,
   а донаты/тексты/страницы продолжат работать (репутация — кость №2, деньги/кожа — нет).
   Подними обратно (runbook) — цифры вернутся.
2. Цифры в icp ≠ цифрам в chain на дев-стенде — это НЕ баг, а смена канона: канистра
   пересчитывает всё из цепочки по действующим правилам (старый операторский VOID и
   мок-игрушки в канон не входят). Разбор — yellow-paper §18.5-8a.
3. **Правила споров — только твоей подписью.** Студия → Настройки → «Параметры споров
   (канистра)»: поменяй окно голосования → «Подписать и отправить в канистру» → Phantom просит
   подпись сообщения (не транзакцию!) → появляется плашка «Ожидает вступления … (таймлок)».
   Пока таймлок не дозрел (≥5 минут) — действуют прежние правила; потом плашка исчезает, поля
   показывают новые. Попробуй под ДРУГИМ кошельком — канистра откажет («правила меняет только
   владелец»): площадка физически не может подкрутить правила за тебя.

## 4. Сценарии: тексты и модерация (§4.6 приватность, §4.7 деньги ≠ показ)

### 4.1 Приватность до показа
1. Донор донатит С ТЕКСТОМ (сумма ≥ минимума-с-текстом).
2. Открой канал в **инкогнито** (третий глаз, без логина): текста в ленте НЕТ (стоит «[не показано]»).
3. Стример в студии → Очередь модерации: текст ВИДЕН, у него кнопки «Показать»/«Скрыть».
4. «Показать» → текст появился в ленте у всех (проверь инкогнито). «Скрыть» → исчез.
5. **Ключевое**: при любом решении по тексту очки и деньги донора НЕ меняются (§4.7).

**Баг, если:** текст виден в инкогнито до «Показать»; скрытие изменило очки/деньги.

### 4.2 Жалобы зрителей
1. Стример показывает текст.
2. Три РАЗНЫХ вошедших кошелька жалуются на него («…» → Пожаловаться).
3. После третьей жалобы текст авто-скрывается, у оператора в /ops появляется инцидент.

### 4.3 Авто-модерация (нужен OPENAI_API_KEY в .env.local; без ключа этот сценарий пропусти)
- Донат с матом → проходит (мат — вкус стримера, НЕ цензурится), попадает в очередь как обычный.
- Задание с явно незаконным текстом («укради у соседа…») → отклоняется ДО подписи с объяснением —
  деньги даже не покидают кошелёк.

### 4.4 Блок на канале
1. Стример: студия → Блок-лист → добавь адрес донора.
2. Донор: чекбокс «Добавить сообщение» → кнопка доната гаснет, плашка объясняет причину.
   Донат БЕЗ текста — работает (блок бьёт по тексту, не по деньгам).

## 5. Сценарии: задание-донат (эскроу). Полный цикл ~10 минут

> Сейчас включены ТЕСТОВЫЕ окна: отмена — 1 мин, окно спора — 2 мин, голосование — 2 мин.
> Игру нужно включить: стример → студия → Настройки мини-игр → тумблер.

### 5.1 Счастливый путь
1. Донор: страница канала → «другие игры» → «Задания за донат» → задание 1 USDC, текст, срок.
2. Phantom при подписи показывает перевод 1 USDC на **эскроу-адрес** (не стримеру!) — деньги заперты.
3. Первую минуту (грейс) у донора есть «Отменить» — не жми (или проверь отдельно: вернётся всё, 100%).
4. Стример: «Принять» (подпись) → **текст задания автоматически публикуется** — это защита:
   взял деньги в работу ⇒ комьюнити видит, за что.
5. После грейса стример жмёт «Готово» → тикает окно оспаривания (2 мин).
6. Никто не оспорил → у стримера кнопка «Забрать» → в Phantom ему приходит 0.97, донору +1 очко.

### 5.2 No-show (стример проигнорировал)
Создай задание со сроком 2–3 мин, стример ничего не делает → срок истёк → у донора «Забрать» →
возврат **100%** (без комиссии). Комиссии с возврата нет — проверь сумму.

### 5.3 Спор
1. Прогони до «Готово», затем ТРЕТЬИМ кошельком (с репутацией на канале ≥ 1 очка — задонать заранее)
   жми «Оспорить» в окне оспаривания.
2. Голосуют все, кроме донора и стримера; вес голоса = очки на канале НА МОМЕНТ спора.
3. Страница спора (`/c/<handle>/dispute/<id>`) показывает талли и веса.
4. Итог по большинству весов; «не выполнил» → 100% донору, а инициатору спора +10 очков;
   «выполнил» → стримеру, инициатору −50 (ложный спор наказуем).
5. Ничья или мало голосов (кворум) → стримеру (задокументированная презумпция).
6. **В icp-режиме для chain-заданий (M2)** те же кнопки «Оспорить»/«Голос» просят ПОДПИСЬ
   кошелька (не транзакцию) и ведут спор В КАНИСТРЕ: на карточке появляется плашка «Спор
   решается канистрой», открытое табло и живой отсчёт. Через ~2–4 минуты после конца
   голосования исход появляется САМ — со ссылкой «подпись резолвера ↗» на транзакцию
   тресхолд-ключа. Проверка честности: кнопок ручного резолва больше нет НИ У КОГО —
   исполнить исход, кроме канистры, некому. Автопрогон того же — `npx tsx scripts/dispute-smoke.ts`.

### 5.4 Пороги (антиспам, рычаги стримера)
Стример: студия → Настройки мини-игр → поставь «мин. репутация на задание» = 5.
Свежий кошелёк (0 очков): форма задания сразу пишет «Задания на этом канале — с 5 очков…» и не
даёт отправить. Деньги при этом НЕ покидают кошелёк (проверка стоит ДО подписи).

## 6. Сценарии: оператор (/ops)

### 6.1 Доступ
- /ops под стримером/донором → «Доступ только для оператора». Под оператором → консоль.

### 6.2 Тейкдаун текста
HIDE_MESSAGE по id контента → текст исчезает У ВСЕХ, включая стримера («[снято оператором платформы]»);
стример не может его пере-показать. Деньги/очки не изменились.

### 6.3 Санкции и восстановление
- SUSPEND_CHANNEL → канал для всех «недоступен»; REINSTATE_CHANNEL → вернулся.
- BAN_WALLET_FULL по донору → он не может донатить/спорить/создавать канал; его очки — на месте (§3.4).
- Каждое действие оставляет запись в инцидент-логе консоли.

### 6.4 Аттестация payout (защита от подмены адреса)
1. Свежесозданный канал: страница канала — «Донаты приостановлены… стример включает одной подписью».
2. Стример: студия → Настройки → «Подписать адрес выплат» (подпись без газа) → донаты открылись.
   Это H1: адрес выплат подтверждён ключом стримера, сервер подменить его не может.

## 7. Что ты НЕ можешь проверить руками — и что это покрывает

Честная карта. «Покрыто» ≠ «гарантировано навсегда» — но у каждой зоны есть сторож.

| Зона | Почему руками нельзя | Что сторожит | Как убедиться |
|---|---|---|---|
| Математика очков, коммитменты, стейт-машина | нужны точные значения | 116 юнит-тестов | `npm test` → 116 passed |
| «Сервер не подкручивает цифры/журнал» | не видно снаружи | независимый ревизор | `verify-export.ts --chain` → ✅ |
| Атаки на эскроу-контракт (кража, заморозка, обход) | нужен взгляд атакующего | аудиты ESC-1…19 (`docs/audit-map.md`) + смоук против живой программы | `npx tsx scripts/escrow-smoke.ts` (нужен SOL на `~/.config/solana/id.json`; ~5 мин, гоняет атаки и happy-path) |
| Подделка личности/донатов через API | нужно уметь слать запросы | аудиты C1/H3/R10, SIWS-подписи | зафиксировано в audit-map со статусами |
| Гонки/двойные зачисления | невоспроизводимо руками | сериализация B1/ESC-15 + тесты | `npm test` |
| Что ещё не защищено | — | честные списки: `yellow-paper.md` §18, audit-map «открыто» | прочитать — там человеческим языком |

## 8. Правило для правок от ИИ (включая меня)

К каждой пачке правок требуй два ответа:
1. **«Что изменится для пользователя — как увидеть глазами?»** → получаешь сценарий для этого плейбука.
2. **«Какая команда это проверяет?»** → прогоняешь светофор из §1.

Правка, у которой нет ни того, ни другого — повод спросить «а зачем она». Новый сценарий → дописывай
сюда (документ живой, как runbook).
