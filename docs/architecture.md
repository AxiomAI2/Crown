# Архитектура: ончейн / оффчейн

Цель документа — зафиксировать, **что где живёт** и **как течёт информация**, чтобы Фаза 2 (бэкенд) и
Фаза 3 (крипта) ложились на уже существующий фронт без переписывания.

---

## 1. Три слоя

```
┌──────────────────────────────────────────────────────────────┐
│ ПРИКЛАДНОЙ СЛОЙ (управляем оператором)                         │
│  фронт, профили, конфиги, тиры/лидерборды, очередь сообщений,  │
│  модерация, баны, дискавери, оверлей, аналитика                │
│  → именно здесь происходит «бан»: оператор перестаёт обслуживать │
└──────────────────────────────────────────────────────────────┘
                 ▲                              ▲
   читает/пишет  │                              │ читает (детерминированно)
                 │                              │
┌────────────────┴───────────┐    ┌────────────┴─────────────────┐
│ ОФФЧЕЙН-ХРАНИЛИЩЕ           │    │ РЕПУТАЦИОННЫЙ ДВИЖОК          │
│ (доверенное, но проверяемое)│    │ pure(journal, config) → rep  │
│  журнал репутации,          │    │  детерминированно,           │
│  тексты сообщений,          │    │  публично перевычислимо      │
│  карантин, конфиги (хэш)    │    └──────────────────────────────┘
└────────────────────────────┘                  ▲
                                                 │ событие DONATION
┌────────────────────────────────────────────────┴─────────────┐
│ ОНЧЕЙН (Solana, пермишенлесс, оператор НЕ касается денег)      │
│  SPL USDC transfer 97%→стример / 3%→трежери + memo, опц. хэш   │
└───────────────────────────────────────────────────────────────┘
```

Ключевая мысль: **деньги и атрибуция — ончейн и пермишенлесс; всё остальное — оффчейн и управляемо.**
Это разделение и есть то, что делает платформу одновременно некастодиальной (деньги) и комплаентной
(контент/доступ).

> Раскладка всего кода и планов по **слоям доверия** (кость/мышцы/кожа), правила каждого слоя и
> исчерпывающий список швов между ними — `trust-layers.md`.

---

## 2. Ончейн (минимум, Фаза 3)

В цепочку кладём только то, что обязано быть необратимым и проверяемым:

- **Транзакция доната** — одна tx с двумя `transfer`-инструкциями: ~97% на ATA стримера, ~3% на ATA
  трежери. Расщепление делает сам фронт.
- **Memo-атрибуция** — `{creator_id, donation_id, msg_ref}` в той же транзакции. По ней индексер
  связывает инфлоу с каналом и сообщением.
- **Опц. хэш текста** — если для конкретного сообщения нужна неизменность пруфа.
- **Сбор активации канала** — отдельный перевод ~$2 USDC на трежери.

Чего в цепочке **нет**: профилей, репутации, текстов, настроек, банов. Всё это оффчейн ради
снимаемости (комплаенс) и дешевизны.

**Самоконтроль атрибуции.** Репутация начисляется только за корректно сформированный донат с
уплаченной комиссией. Не заплатил трежери → движок такой инфлоу не засчитывает. Это убирает стимул
обходить комиссию.

---

## 3. Оффчейн (Фаза 2)

**Журнал репутации** — апенд-онли лог событий (схема — `data-model.md`). Репутация = свёртка журнала.
Журнал — единственный источник истины по репутации; текущее число всегда можно перевычислить.

**Репутационный движок** — чистая функция `reputation(journal, config) → points`. Без сайд-эффектов,
без обращения к сети внутри. Та же функция запускается и на сервере, и (теоретически) у любого, кто
скачал журнал. Это инвариант: **сервер считает по открытой формуле, а не «решает».**

**Тексты сообщений** — в модерируемом хранилище, снимаемые. Карантин — закрытый бакет с retention.

**Конфиги стримера** — версионируются и хэшируются. Каждое событие в журнале помнит `config_version`,
по которому были забанкованы очки → смена формулы не переписывает прошлое.

---

## 4. Поток данных: жизнь одного доната

```
1. Донор жмёт «Задонатить» на странице канала, вводит сумму (+опц. текст).
2. Фронт валидирует: сумма ≥ max(газовый флор, минимум стримера); текст ≤ лимит; локальный wordlist.
3. [Фаза 3] Фронт собирает одну tx (97/3 + memo). Кошелёк подписывает. Сеть финализирует.
   [Фаза 1/2] DataProvider имитирует/принимает донат и возвращает запись.
4. Индексер ловит инфлоу в трежери, парсит memo → создаёт событие DONATION в журнале.
5. Движок начисляет очки СРАЗУ (banked по текущему config_version). Деньги уже у стримера.
6. Текст → HELD (видят только стример + модераторы) → модерационный конвейер.
7. Стример/модератор: «Показать» → SHOWN (оверлей/фид) ИЛИ «Скрыть» → HIDDEN.
   В обоих случаях деньги и репутация донора остаются.
```

Обрати внимание: шаги 5 и 6 независимы. Деньги/репутация не ждут модерацию; модерация решает только
судьбу **текста**.

---

## 5. Граница «трастлесс» vs «управляемо» (частая путаница)

Нет противоречия между «репутация трастлесс» и «оператор может банить»:

- **Учёт репутации — трастлесс.** Кто угодно скачивает ончейн-донаты + опубликованный конфиг и
  получает ту же цифру. Оператор не может втихую накрутить кому-то репутацию.
- **Доступ к платформе — управляем.** Оператор законно решает, кого обслуживать (бан = перестать
  отдавать страницу/оверлей/индексацию). Деньги при этом остаются пермишенлесс — забаненный может
  слать сырые ончейн-переводы, но это уже не «донат» (ни сообщения, ни репутации, ни аудитории).

---

## 6. Что это значит для фронта (Фаза 1)

Фронт не должен знать, откуда данные. Он знает только интерфейс `DataProvider`
(`../frontend/mock-data.md`). Маппинг слоёв на реализации:

| Метод DataProvider | Фаза 1 (mock) | Фаза 2 (api) | Фаза 3 (chain) |
|--------------------|---------------|--------------|----------------|
| чтение репутации/лидерборда | фикстуры | бэкенд (из движка) | индексер → движок |
| отправка доната | имитация успеха | API записывает | кошелёк подписывает tx |
| очередь модерации, show/hide | in-memory | API + хранилище | то же, что Фаза 2 |
| конфиг канала, тиры | фикстуры | API | API (конфиг оффчейн) |
| подключение/сессия | мок-адрес | мок/реал | wallet-adapter + SIWS |

Поэтому деньги и крипта — последняя фаза: они меняют **реализацию** методов отправки доната и чтения
репутации, но не сигнатуры и не экраны.
