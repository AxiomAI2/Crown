# ADR 0016 — Мини-игры как модули (реестр + включение на канале)

**Статус:** принято (2026-06-29)
**Контекст:** строим первую мини-игру (ADR 0015 — шов с репутацией). Игр будет несколько (спека
`docs/games/escrow-task-spec.md` §4: «конкретные игры подключаются модулями»). Нужна структура, при
которой игры подключаемы, ядро остаётся чистым, а сайт адаптируется без хардкода под каждую игру.

## Проблема

Если лепить первую игру прямо в ядро (`components/domain/`, экраны, плоский `DataProvider`), то вторая и
третья потребуют переписывания: разрастётся интерфейс данных, страница канала и студия обрастут
`if (game === ...)`, игровой код смешается с ядровым. Нужно зафиксировать каркас ОДИН раз.

## Решение

**1. Каждая игра — самодостаточный модуль** в `src/games/<id>/`: манифест + свои компоненты, хуки,
логика (стейт-машина на моке), типы. Ядро (`lib/`, `components/domain/`, экраны ядра) про конкретные
игры НЕ знает — оно работает с репутацией и журналом (ADR 0015), игры стоят сверху.

**2. Реестр** `src/games/registry.ts` — единственный список игр. Добавить игру = добавить папку + одну
строку в реестр. Никаких других правок ядра.

**3. Манифест** `GameModule` (`src/games/types.ts`) — данные-онли: `id`, `title`, `tagline`, `status`,
`specDoc`. `status: "building"` — игра видна в доках/деве, но каналам к включению не предлагается;
`"available"` — можно включать. Слоты под экраны (панель канала / студии) добавятся в манифест, когда
появятся компоненты (G1), чтобы не плодить пустых заглушек.

**4. Включение — на канале.** `enabledGames: GameId[]` в `ChannelConfig` (спека §10 «какие мини-игры
включены на канале»); по умолчанию пусто — cold-start: стример включает игру, когда комьюнити накопило
репутацию (спека §8). Студия получает секцию «Мини-игры» — список из реестра с тумблерами.

**5. Сайт рендерит из реестра.** Страница канала и студия ИТЕРИРУЮТ реестр (по включённым на канале), а
не хардкодят игру. Новая игра появляется в UI автоматически.

**6. Данные — один game-bus в `DataProvider`.** Вместо ~6 методов на каждую игру (раздувание
интерфейса и RPC-кодека) — две обобщённые точки: `gameAction(input)` (мутации) и `gameQuery(input)`
(чтения), где `input = { gameId, channelId, op, payload }`. Каждый модуль валидирует/обрабатывает свои
операции; типобезопасность восстанавливается типизированными хуками ВНУТРИ модуля. Реализуется
mock → api → chain, как всё остальное; экраны чейн не зовут.

## Последствия

- **Плюс:** ядро остаётся чистым; игры подключаемы папкой; сайт адаптируется сам; core-интерфейс и
  RPC-кодек не пухнут с каждой игрой; репутационный шов уже отдельно зафиксирован (ADR 0015).
- **Минус (честно):** game-bus теряет статическую типобезопасность НА ГРАНИЦЕ (payload — `unknown`) —
  компенсируем типизированными обёртками-хуками в модуле и валидацией на входе. Реестр — ручная
  регистрация (это и плюс: явный, видимый список).
- `GameId` — объединение строк; добавление игры расширяет его и ловится типами во всех местах разбора.

Не меняет инварианты §4 — они закрыты ADR 0015. Этот ADR только про ФОРМУ кода и сайта под много игр.
