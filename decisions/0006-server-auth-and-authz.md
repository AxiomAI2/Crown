# ADR 0006 — Серверная аутентификация (SIWS) и авторизация мутаций

- **Статус:** принято
- **Дата:** 2026-06-20
- **Контекст:** аудит безопасности показал, что личностью запроса был **непроверенный** `body.address`
  (ADR 0005 §1), а «SIWS» в `ChainDataProvider.connect` подписывал сообщение и **выбрасывал** подпись —
  сервер её не проверял. Плюс мутации стора не проверяли права. Следствия (все воспроизведены `curl`):
  любой мог выдать себя за оператора и банить каналы / делать `ADMIN_VOID` (репутация — продукт), править
  конфиг/блок-лист любого канала, публиковать приватный текст доноров (нарушение инварианта §4.6),
  `__reset` стирал весь стор одним запросом, а `createChannel` принимал произвольный `payoutAddress`
  (имперсонация стримера → увод донатов). Это **замена** модели личности из ADR 0005 §1.

## Решения

### 1. Личность = проверенный токен, а не строка из тела
SIWS-поток реализован по-настоящему (`src/server/auth.ts`, общий формат сообщения — `src/lib/chain/siws.ts`):
1. `__authNonce(address)` — сервер выдаёт **одноразовый** nonce (24 байта) с TTL 5 мин и каноническое сообщение.
2. кошелёк подписывает сообщение (`signMessage`), клиент шлёт base64-подпись.
3. `__authVerify(address, sig)` — сервер проверяет ed25519-подпись над тем же сообщением (встроенный
   `node:crypto`, без новых зависимостей: сырой 32-байтный pubkey оборачиваем в SPKI DER), гасит nonce,
   выдаёт session-токен (TTL 12 ч).
4. RPC несёт `token`; `resolveToken` → проверенный `address`. **`body.address` в проде игнорируется.**

Клиент: `ChainDataProvider.ensureAuth()` ведёт вход идемпотентно (повторно не подписывает; переиспользует
непросроченный токен из `localStorage`), запускается из `ChainWalletBridge` после подключения кошелька.
Донаты **не требуют** входа (деньги едут ончейн, приём — трастлесс), вход нужен для действий по личности.

### 2. Авторизация на каждой мутации (стор — единая точка истины)
В `MockDataProvider` добавлены `requireSession / requireOperator / requireChannelOwner / requireChannelManager`:
- оператор-действия (`applyOperatorAction`, `getOperatorQueue`, `getIncidentLog`) — только `OPERATOR_ADDRESS`;
- конфиг и активация канала — только владелец;
- модерация (`getModerationQueue`, `setMessageState`) и блок-лист — владелец или модератор канала
  (бан-операции требуют скоуп `queue_and_block`);
- приватный текст редактируется и в публичном `listDonations` (виден только менеджерам канала, инвариант §4.6).

Так как `isOperator` теперь key-gated (нужен приватный ключ для подписи), публичность адреса оператора
больше не даёт прав — но разнести роли оператора и трежери всё равно стоит (см. ограничения).

### 3. Dev-методы и dev-вход — только вне прода
`__reset` убран из сетевого белого списка и обслуживается отдельной веткой только при `NODE_ENV !== production`.
Dev-вход по адресу без подписи (`body.address`, тулбар/`/connect` под mock/api) тоже работает **только вне
прода**; в проде личность даёт исключительно проверенный токен.

### 4. Валидация входов на денежном пути
`createChannel` проверяет `payoutAddress` (формат base58 без web3.js — `isLikelyBase58Address`, чтобы не
тащить Solana в bundle mock/api) и `handle` (`^[a-z0-9_]{3,32}$`); уникальность handle — **без учёта
регистра** (анти-имперсонация `@Brand`/`@brand`).

## Проверено / не проверено

**Проверено (headless, прод-сборка `next start`, `curl` + ed25519-скрипт):**
- спуф оператора через `body.address` без токена → `NO_SESSION`; `__reset` → `BAD_METHOD`; конфиг/очередь
  модерации без токена → `NO_SESSION`; публичный `listChannels` работает;
- полный SIWS-раунд реальным ключом → токен; **реплей** одноразового nonce → `AUTH_FAILED`; **подделка**
  подписи → `AUTH_FAILED`; мусорный токен → аноним;
- авторизованный не-оператор зовёт `applyOperatorAction` → `FORBIDDEN`; авторизованный `createChannel` → ок;
- валидация: кривой payout → `BAD_PAYOUT`, кривой handle → `BAD_HANDLE`, регистровый сквоттинг → `HANDLE_TAKEN`;
- `next build` (chain) зелёный, typecheck чист.

**НЕ проверено (нужен браузер/кошелёк):** реальная подпись Phantom/Solflare и UX-поток входа в браузере.

## Честные ограничения
- Сторы nonce/сессий — in-memory на `globalThis` (как и сам стор): не переживают рестарт и не шарятся между
  инстансами. В проде переезжают в общий слой персистентности (Postgres/Redis) вместе со стором.
- Токен в `localStorage` уязвим к XSS (стандартный компромисс session-токена); ротация по TTL 12 ч.
- Оператор по умолчанию = трежери (`OPERATOR_ADDRESS ?? TREASURY_OWNER`). Перед mainnet развести роли
  (отдельный, желательно мульти-сиг, оператор) и увести ключи из плейнтекст-файла в KMS/HSM.
- `createChannel` подтверждает владение **адресом-плательщиком** (подпись на входе), но не владение
  **handle/брендом**. Доказательство прав на имя (верификация соц-аккаунта) — отдельный продуктовый шаг;
  до него донору в UI стоит показывать статус «канал не верифицирован».
