# Crypto / On-chain — спецификация (Фаза 3)

Финальная фаза: реальные деньги на Solana, реальный кошелёк, реальная некастодиальность. Реализует
ту же `DataProvider` (`frontend/mock-data.md` §1) — реализацию `ChainDataProvider` + индексер. Экраны
не меняются; меняется то, что стоит за `connect()`, `createDonation()` и чтением донатов.

> **До хостинга хотя бы одного чужого доллара на мейннете — живая юридическая консультация в выбранной
> юрисдикции** (`docs/legal-and-risk.md`, мастер-переменная). Эта спека — про механику, не про допуск.

---

## 1. Стек

- **Кошелёк/подключение:** `@solana/wallet-adapter-react` (+ `@solana/wallet-adapter-react-ui`).
  Построен на Wallet Standard; даёт `connect/disconnect/signMessage/signTransaction`.
- **Аутентификация:** **SIWS (Sign-In-With-Solana)** — `authorize` + `signMessage` одной операцией,
  без газа. Подпись = логин. Бэкенд верифицирует подпись адреса → сессия.
- **Транзакции/чтение:** `@solana/kit` (бывш. web3.js v2) или дружелюбная обёртка `gill` (+`@gillsdk/react`).
  Нужны стандартные программы: **SPL Token** (перевод USDC), **Associated Token Account** (ATA),
  **Memo**. Anchor не требуется (кастомных программ в ядре нет).
- **Мост типов:** wallet-adapter завязан на web3.js v1; для kit использовать `@solana/compat` или Wallet
  Standard-хуки `@solana/react`. Зафиксировать один путь в начале фазы.

Сверить актуальность версий перед стройкой — экосистема Solana движется быстро.

## 2. Константы сети

```
USDC mint (mainnet): EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v
Treasury ATA:        <адрес USDC-ATA трежери платформы>   // куда идёт 3%
Commitment:          "confirmed" для UX, "finalized" для зачёта репутации индексером
```

Сначала всё калибруется на **devnet** (свой devnet USDC mint), потом mainnet.

## 3. Донат-транзакция (сердце фазы)

**Одна транзакция**, собирается фронтом, подписывается донором. Деньги идут **напрямую**:
донор → стример (97%) и донор → трежери (3%). Оператор средства доната **не трогает** (некастодиальность,
инвариант §4.1). Инструкции:

1. (если нужно) **CreateATA** для ATA стримера, если его ещё нет — оплачивает донор (rent ~0.002 SOL).
2. **SPL Transfer** `net` (97%) USDC: ATA донора → ATA стримера (`payoutAddress`).
3. **SPL Transfer** `fee` (3%) USDC: ATA донора → ATA трежери.
4. **Memo** с атрибуцией: `{ creator_id, donation_id, msg_ref }` (компактный JSON/CBOR).

```ts
// псевдо-сборка
const ix = [
  ...(streamerAtaMissing ? [createAtaIx(streamerAta, payoutAddress, USDC)] : []),
  transferCheckedIx(donorAta, streamerAta, donor, net, USDC, 6),
  transferCheckedIx(donorAta, treasuryAta, donor, fee, USDC, 6),
  memoIx(JSON.stringify({ c: creatorId, d: donationId, m: msgRef ?? null })),
];
// recentBlockhash + feePayer = donor; priority fee при заторе; затем signTransaction
```

- `fee = round(amount * 0.03)`, `net = amount - fee` — целочисленно в micro-USDC (6 знаков). Те же числа,
  что считал бэкенд/мок.
- **Газ:** донору нужен ~0.001 SOL на подпись; при заторе добавлять **priority fee**; **газовый флор** —
  минимальный донат растёт с газом (динамический минимум на фронте, чтобы донат имел смысл).
- **Самоконтроль комиссии:** репутация начисляется **только** за корректно сформированный донат с
  уплаченной комиссией. Не заплатил трежери (нет инструкции 3) → индексер такой перевод **не видит**
  как донат → нет репутации. Это делает обход комиссии бессмысленным.
- `donation_id`/`msg_ref` генерятся клиентом до подписи и кладутся в memo, чтобы индексер связал
  ончейн-инфлоу с оффчейн-текстом.

## 4. Индексер (ончейн → журнал)

Отдельный сервис. **Истина о деньгах — цепочка, не клиент.** Поток:

```
наблюдать инфлоу в Treasury ATA (3%-нога)
  ▼
по сигнатуре tx достать инструкции → проверить парность:
   transfer→streamer (97%) + transfer→treasury (3%) + memo
  ▼
распарсить memo {creator_id, donation_id, msg_ref}; сверить пропорции и адрес payout канала
  ▼
дождаться "finalized"
  ▼
идемпотентно записать Donation(final:true) + LedgerEvent(DONATION) с tx_signature
   (bankPoints по конфигу на момент tx — банкинг)
  ▼
если msg_ref → принять оффчейн-текст (через API), state HELD → модерация (backend §7)
```

- **Идемпотентность по `tx_signature`** — повтор обработки не дублирует событие.
- **Что не парное / без комиссии / без memo** — игнор (сырой перевод ≠ донат: ни репутации, ни текста,
  ни аудитории — `docs/core-spec.md` §10).
- Репутация зачитывается по **finalized**, чтобы не учитывать откатившиеся.
- Шаг 3 донат-флоу из `backend/spec.md` §6 в этой фазе запускает **индексер**, не клиентский запрос.

## 5. Активационный сбор

`activateChannel`: одноразовый перевод ~$2 USDC на трежери. Индексер ловит инфлоу с memo
`{ activate: channel_id }` → `BASIC → ACTIVE`. **Сбор, не залог:** возвратный залог у оператора вернул бы
кастодиальный/MT-риск (`docs/legal-and-risk.md`); если нужна возвратность — только некастодиальным
локом в контракте (вне ядра).

---

## 6. Опциональный ончейн-якорь текста

Если нужна неизменяемость пруфа — класть `content_hash` сообщения в memo/отдельную инструкцию.
Сам текст остаётся оффчейн и снимаемым (для комплаенса). По умолчанию в ядре можно не якорить.

## 7. `ChainDataProvider` (фронт)

Реализует интерфейс §1, но:
- `connect()` → wallet-adapter + SIWS; `getSession()` из подписанной сессии.
- `createDonation()` → собирает и шлёт транзакцию §3 через кошелёк; возвращает оптимистичный
  `DonationResult`, финальный зачёт подтверждает индексер (UI показывает «подтверждается → подтверждён»).
- Чтение (`getStanding/listDonations/leaderboard/...`) — через бэкенд-API, наполняемый индексером
  (гибрид: запись инициируется ончейн, чтение — из индексированной БД для скорости).
- Конфиг/модерация/баны — те же API, что в Фазе 2 (оффчейн-слой оператора).

Переключение: `NEXT_PUBLIC_DATA_SOURCE=chain`. Экраны и флоу (`user-flows.md`) — те же.

## 8. Калибровка на devnet (до mainnet)

- Реальные суммы газа, priority fee, газовый флор минимального доната.
- Стоимость и тайминг создания ATA стримера в той же транзакции.
- Поведение индексера под бёрстом (флуд самодонатами — главная угроза, `docs/core-spec.md` §10).
- Калибровка анти-флуда: размер сбора активации, rate-limit, retention, скорость карантина/бана.
- Калибровка репутации: кривая (linear/bracket), decay.

---

## 9. Инварианты, которые легко сломать именно здесь

1. **Некастодиальность.** Никакой инструкции/программы/ключа, дающего оператору власть над чужими
   средствами. Трежери получает только свою 3%-ногу как получатель перевода, не как кастодиан доната.
2. **Деньги финальны.** Никаких механик возврата ончейн. (Самодонат-флудер «возвращает» себе сам —
   это его проблема и его публичный самодонос.)
3. **Репутация непередаваема.** Не токенизировать репутацию ни при каких удобствах. Никаких NFT/SPL
   «бейджей», которые можно продать — это сломает юридический замок «не ценная бумага».

---

## Критерии готовности Фазы 3

- Донат проходит как одна tx (97/3 + memo + ATA-при-необходимости) на devnet, затем mainnet.
- Индексер идемпотентно строит журнал из finalized-инфлоу; цифры репутации совпадают с движком §3 backend.
- SIWS-логин работает; сессии привязаны к подписи.
- Активация через ончейн-сбор; газовый флор и priority fee откалиброваны.
- Все три инварианта §9 проверены ревью; пройдена юр-консультация по мастер-переменной до mainnet.
